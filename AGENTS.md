# AGENTS.md — EventOps Superpowers Rules

## 0) 항상 읽기
- 이 폴더에서 코딩, 수정, 파일 생성 작업을 시작하기 전에 항상 이 `AGENTS.md`를 먼저 읽고 따른다.
- 상위 폴더 규칙과 함께 적용하되, 충돌 시 이 파일을 우선한다.

## 1) 문서 확인 순서
- 이 폴더에 로컬 문서가 생기면 먼저 확인한다.
- 로컬 문서가 없으면 상위 프로젝트 문서를 기준으로 작업한다.
  - `../README.md`
  - `../QUICKSTART.md`
  - `../docs/PRD.md`
  - `../docs/AGENT_FEATURE_PLANNING.md`
- 계획성 요청이면 구현 전에 PRD/Planning 관점으로 정리한다.

## 2) Superpowers 기본 적용 원칙
- 작업 시작 전 관련 스킬 존재 여부를 먼저 확인한다.
- 1%라도 관련 가능성이 있으면 스킬 사용을 우선한다.
- "단순 작업이니 스킬 불필요"라고 가정하지 않는다.
- 추측보다 증거, 즉흥 대응보다 절차를 우선한다.

## 3) 필수 워크플로우 순서
1. `brainstorming`
   - 바로 코드를 쓰지 말고 접근법 2~3개, 리스크, 선택 기준을 먼저 정리한다.
2. `writing-plans`
   - 작업을 2~5분 단위로 쪼개고, 파일 경로/산출물/검증 명령/롤백 포인트를 적는다.
3. `subagent-driven-development`
   - 병렬화 가능한 작업은 서브에이전트 단위로 나누고, 결과는 스펙 준수와 코드 품질 두 단계로 검토한다.
4. `test-driven-development`
   - RED → GREEN → REFACTOR 순서를 지킨다.
   - 테스트 없이 프로덕션 코드만 추가하지 않는다.
5. `systematic-debugging`
   - 버그는 근본 원인 조사 후 수정한다.
   - 같은 문제를 3회 이상 수정 실패하면 구조적 원인을 다시 검토한다.

## 4) 구현 수칙
- 기능 변경 시 문서/코드/테스트를 함께 갱신한다.
- 민감정보(API Key, 토큰, 비밀번호)는 코드/문서/로그에 평문으로 남기지 않는다.
- 사용자 메시지와 내부 디버깅 정보는 분리한다.
- 상위 프로젝트의 설정 키 이름과 의미를 임의로 어긋나게 바꾸지 않는다.

## 5) 검증 수칙
- 검증은 가능하면 Docker 기준으로 재현 가능해야 한다.
- 테스트 또는 검증을 생략했다면 이유와 남은 리스크를 명시한다.
- 완료 보고에는 최소한 다음을 남긴다.
  - 변경 내용
  - 실행한 검증
  - 미실행 검증과 이유

## 6) 폴더 운영 원칙
- 이 폴더에 새 문서나 스크립트를 추가하면 이후 작업에서는 그 문서를 우선 기준으로 삼는다.
- 상위 프로젝트 규칙과 충돌하면 더 구체적인 로컬 규칙을 우선하고, 충돌 사실을 작업 결과에 명시한다.
